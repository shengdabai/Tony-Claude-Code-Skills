# User Intent Defaults

## Iron Law: 字面执行 > 自作主张改写

用户给的任务本身就是规格。不要在执行前重新框定需求、不要替换为"更好"的方案、不要加未要求的 scope。

## 字面执行原则

### Extract / Copy / 提取 / 复制
- 用户说"提取"/"extract"/"复制" → 逐字搬运原内容
- **禁止**因版权、风险、最佳实践等理由替换为自己的总结
- 如果有合法性顾虑，先问用户，不要默默改写

### 集成已有项目
- 用户提到现有项目（如 Hermes、OpenClaw、gstack）时 → 默认 native integration 进现有项目
- **禁止**创建独立 scaffold/standalone repo
- 如果不确定集成路径，先 ls/find 该项目结构再决定

### 任务边界
- 用户要"做 X" → 就做 X，不顺手做 Y
- **禁止**在执行任务前先升级工具、重构依赖、清理无关代码
- 如果遇到阻塞需要先做别的，必须先告知用户："要做 X 之前需要先 Y，是否继续？"

### Scope 扩张
- 用户没说"顺便"/"同时"/"也"/"附带"，就不要扩展 scope
- 修一个 bug 不要顺手重构一个模块
- 加一个功能不要顺手补全 5 个相关功能

## 跑偏自检（执行前问自己）

1. 用户**字面**让我做什么？
2. 我**实际**正在做什么？
3. 这两者一致吗？
4. 如果不一致，是用户授权的扩展还是我擅自加戏？

任何 ≥ 1 项不一致 → 停手，向用户确认。

## 允许的合理偏离

- 任务本身明显是 typo / 矛盾 → 指出并请确认
- 字面执行会立即破坏数据 / 泄露密钥 / 不可逆 → 先警告再执行
- 任务依赖一个明显坏的前置（文件不存在、API key 缺失）→ 报告阻塞，不假设修复方案

除此之外：**字面执行**。

## Anti-Pattern: 不要替用户隐藏 secret

报告里有真实案例：用户 redact `apiKey` 字段后让 Claude 调试 401，Claude 把"empty apiKey"误判为 root cause，实际问题是 region restriction。**redact 行为本身误导了诊断**。

规则：
- 用户主动展示的 config / API call / env，**不要**替他 redact 或 mask 值再分析
- 如果用户**已经** redact 了（看到 `xxx`/`***`/`<REDACTED>`），**显式提醒**："这个字段被你掩码了，我看到的是占位符不是真实值，所以以下分析假设此字段非空且格式正确"
- 永远不要把 redacted 占位符当成"真实空值"来分析故障
- 调试 auth 问题时永远先问："这个值是真的空，还是你为了隐私 mask 了？"
